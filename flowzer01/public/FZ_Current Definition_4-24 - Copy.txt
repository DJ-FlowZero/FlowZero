# FlowZero – Daily Specification (v0.1 Draft)

## 1. Overview

FlowZero is a lightweight, human-centric system for managing meaning, context, and action in everyday life. It enables individuals and small groups to tune their interaction with sytems using simple constructs that remain readable, flexible, and composable.

At its core, FlowZero solves a common problem:

> Humans operate in rich, contextual environments, but traditional systems force rigid structure or lose meaning entirely.

FlowZero provides a middle path:

* **natural language remains intact**
* **meaning is explicitly anchored**
* **context is dynamically applied**

This is achieved through three core primitives:

* **Profiles (context)**
* **Nomers (meaning anchors)**
* **Stickies (situational modifiers)**

Together, these form a minimal yet powerful system capable of handling a wide range of real-world coordination tasks, by using portable "Modes" that fine tune receiving systems.

---

## 2. Core Concepts

### 2.1 Profiles

A **Profile** represents any identifiable entity:

* Person (Chris, Pia, Uli)
* Place (Dachau, Allach, Practice Room)
* Object (Mitch guitar, generator, car)
* Environment (restaurant, building, gig setup)

Profiles serve as containers for:

* Nomers
* JAR definitions
* Norms (rules and conventions)
* Contextual knowledge

Profiles define:

> **Where and for whom meaning applies**

---

### 2.2 Nomers

A **Nomer** is a labeled handle that represents a concept with a defined meaning.

Nomers can be:

* existing words with specialized meaning (“garlic”)
* nicknames (“Mitch”)
* acronyms (“EMC”)
* invented terms (“foodeldunk”)
* phrases (“the usual”)

A nomer is not defined by its form, but by its function:

> A nomer is a **locally fixed meaning for a handle**

Nomers are written in square brackets:

```text
[Mitch]
[Dachau]
[Breakfast]
```

---

### 2.3 Tokens

A **Token** is the system-level representation of a nomer.

* Always enclosed in square brackets
* Used for parsing and recognition

```text
[Mitch]
```

A token represents a nomer but is not the meaning itself.

---

### 2.4 JAR (JSON Accessible Render)

A **JAR** is the structured meaning package behind a nomer.

It can contain:

* definition
* properties
* relationships to other nomers
* media references (images, links)
* usage notes and quirks

Example:

```json
{
  "Mitch": {
    "type": "guitar",
    "description": "Fender Telecaster, Hurricane Mitch mahogany",
    "tools": ["2.5mm hex", "long trem arm"],
    "notes": "slightly sensitive tuning, expressive tone"
  }
}
```

Conceptually:

* label = outside of the jar
* JAR = everything inside

JARs are:

* structured
* portable
* expandable on demand

---

### 2.5 Stickies

**Stickies** represent situational context.

They:

* modify nomers or profiles
* override defaults
* are temporary and dynamic

Example:

```text
[Breakfast] (for 3, at [Dachau], skip eggs)
```

Stickies allow the system to express:

> “What is different today”

---


* system-recognized meaning

---

### 3.2 Error / Unresolved Nomer

* nomer recognized

## 4. Semantic Fields

FlowZero distinguishes between three semantic layers:

### Public

* widely shared meaning
* accessible to all
* example: legal, technical, sports terminology

### Private

* shared within a group
* requires shared nomerpedia
* example: family shorthand, band context

### Secret

* restricted meaning
* not exposed without permission
* example: passwords, medical data, safe words

Default behavior:

* unknown or sensitive meaning is not expanded
* system favors non-disclosure over incorrect inference

xExplainFrame: A concise, familiar analogy used to make complex or abstract concepts understandable in everyday language. It maps structure to common experience for quick intuition, without claiming exact equivalence or replacing formal definitions.

[Digital Air] can be understood as an ambient computing system that comprises all accessible processing but hides the
complexity of the underlying system architecture. The most common Digital Air implementation today is the accessibility of knowledge behind a Google Search, or access to an LLM. Digital Air may, when properly used and configured, appear like a game engine, but instead of enforcing fixed rules, it converges around real-world observations, physics, and statistical patterns. Unlike traditional game engines that can override reality (e.g. anti-gravity, resurrection), Digital Air operates as a probabilistic environment shaped by plausibility, where outcomes emerge through convergence rather than strict rule execution. Within this frame, CERO acts as the game controller, selecting and tuning Flow Modes (FM), shaping the active context, and guiding interaction so the system behaves coherently and predictably for the user. FlowZero therefore provides a controllable interface to an otherwise fluid system—making Digital Air feel interactive, responsive, and stateful, while remaining grounded in real-world constraints.


[Game Analogy]: Digital Air may appear like a game engine, but it converges around real-world constraints rather than enforcing arbitrary rules. FlowZero provides a familiar interaction layer: Flow Modes act like avatar configurations, the Flexicon defines interpretation rules, PUCK carries session context, and CERO serves as the controller and boundary to execution.
One-liner: Digital Air appears like a game engine; FlowZero provides the controller and configuration layer for interacting with it under real-world constraints.

[Gamer's Fallacy]: The Gamer's Fallacy (GF) is a type of Paradigm Frame Lock (PFL) where expectations shaped by highly responsive, configurable environments are unconsciously applied to the real world. These environments can include interactive gaming systems (where outcomes are controllable and failures are reversible) as well as pampered or high-service contexts (e.g. service staff, concierge, or butler-style support) where individual needs are consistently anticipated and fulfilled. Under GF, people may expect high configurability, low-cost failure, and that systems or others will adapt to their personal preferences. In real-world contexts—governed by shared constraints, policies, and non-negotiable trade-offs—this leads to friction. In edge cases, GF can manifest as entitled behavior, where individuals expect others to adjust to their needs, quirks, and idiosyncrasies without reciprocal adaptation.
One-liner: Gamer’s Fallacy = expecting real-world systems to behave like configurable, low-risk environments.

[Differential Gearbox]: FlowZero can be understood as a differential gearbox with traction control for intent. In mechanical systems, a differential allows multiple wheels to rotate at different speeds while remaining part of the same system, and traction control prevents loss of grip under varying conditions. Similarly, FlowZero aligns interactions across human (slow), social (medium), and digital (fast) layers, allowing each to operate at its natural pace while maintaining coherence. The system does not change the underlying components but matches their operating speeds and conditions. Traction control corresponds to mechanisms like Flexicon constraints, PUCK context, and CERO intervention, which prevent misalignment, overreach, or unrealistic expectations (e.g. Gamer’s Fallacy), ensuring that intent translates into effective action under real-world constraints.
One-liner: FlowZero = differential gearbox with traction control for intent.

[Neighborhood Concierge]: FlowZero can be applied as a lightweight neighborhood concierge system for assisting someone with recurring, preference-sensitive tasks such as shopping. A Profile is created for the person (even without full identity, but optionally with a user-chosen nickname such as MahjongLady2000), and a Flexicon captures their specific meanings for common items (e.g. “milk” → 2% low fat from Rewe, “butter” → preferred brand and type). This allows simple requests to be interpreted consistently without repeated clarification. Changes in condition (e.g. dietary restrictions, temporary needs) are expressed as Stickies, modifying behavior without altering the underlying definitions. The Profile, Flexicon, and Stickies can be packaged into a PUCK and shared with Digital Air, allowing AI systems to generate aligned outputs (e.g. “create a shopping list within a €50 budget”) that respect personal preferences and constraints. The result is a system where everyday help can be provided reliably, whether executed manually or assisted by AI.

One-liner: Shared shorthand + personal preferences + PUCK → reliable, AI-assisted everyday assistance.

[Stammkunden-Affordance]: The Stammkunden-Affordance (SKA) describes the effect where a system, person, or service behaves like a well-trained regular-service provider (“they know you”) by aligning with a user’s established preferences, shorthand, and context. Instead of requiring explicit instructions each time, the system interprets intent based on prior knowledge encoded in Profiles, Flexicon, and PUCKs. SKA is not based on identity exposure, but on consistent, portable context that allows the system to act as an expert on user intent. It reflects the familiar experience of being a regular customer, where minimal input yields highly aligned outcomes. In FlowZero, SKA emerges when Digital Air is properly primed with a user’s Mode, enabling responsive, context-aware interaction without repeated explanation.
One-liner: Stammkunden-Affordance = the system knows your “usual” without you having to explain it.

[CCML Minimal Core]: CCML can be understood as a minimal symbolic “control panel” for intent, using a fixed set of primitives (7 max) to encode meaning, context, and execution in a compact, human-readable form. Each symbol has a single, stable role: tokens represent meaning, types classify objects, flags constrain behavior, macros activate modes, and stickies capture what is different right now. This allows complex situations to be expressed as short, composable statements that can be interpreted consistently by humans and systems. The design intentionally limits the number of primitives to prevent drift and encourage reuse, similar to how small instruction sets enable powerful computation.
One-liner: CCML = a 7-symbol control panel for expressing intent, context, and action.
[CCML Syntax]: CCML uses a minimal set of 7 primitives to encode intent, context, and execution in natural text.

[token]     → nomer (Flexicon-resolved meaning handle)  
{type}      → object classification (profile / sticky context)  
<flag>      → processing or constraint modifier (e.g. public, private, no_dairy)  
@macro      → activation key (invokes Flow Mode / PUCK / macro)  
!token      → unresolved, protected, or flagged nomer  
//sticky    → inline situational modifier (temporary override)  
::          → binding / association operator (minimal linkage)

Rules:
- Each primitive has a single, stable role  
- Tokens are semantic only; control is expressed via flags, macros, and stickies  
- Syntax is human-readable and machine-parsable  
- Expansion of meaning is handled via Flexicon, not syntax  
- Composition is preferred over nesting or complex grammar

One-liner: CCML syntax = 7 primitives, each with one job, combined to express intent clearly.

[Münchhausen Architecture Principle]: The Münchhausen Architecture Principle (MAP) describes a system design approach in which the system is sufficiently self-describing to enable its own reconstruction from within (bootstrap). All essential meaning, structure, and behavior are encoded in portable, human-readable artifacts such as Profiles, Flexicon (nomers), and Stickies. Given this description and access to the stored data, a sufficiently capable AI/LLM can reconstruct a minimal working version of the system and its affordances without relying on fixed implementation details. The system effectively “pulls itself up by its own bootstraps,” not through hidden mechanics, but through explicit, structured descriptions of how it operates.
One-liner: MAP = the system contains enough description to rebuild itself from its own semantic artifacts.
[MAP] + [Dictionary-to-Sonnet]: The Münchhausen Architecture Principle can be understood like having a complete dictionary and asking a system to write a sonnet. The dictionary contains all the building blocks (words, meanings, relationships), but not the final artifact. Given sufficient structure and context, a capable system can generate new, functional outputs from those primitives. Similarly, FlowZero encodes meaning in Profiles, Flexicon, and Stickies so that, even without fixed implementation, a system can reconstruct useful behavior and affordances from the available semantic components.
One-liner: Given the building blocks and rules, the system can recreate the outcome.

[Digital 1 vs Digital 2]: Digital 1 systems infer user intent by observing behavior and applying statistical models based on standard patterns. Digital 2 systems align with user intent by tuning behavior based on explicitly provided context (Profiles, Flexicon, Stickies, PUCK). Instead of guessing from past actions, the system responds to what the user chooses to reveal in the current situation. FlowZero focuses on optimizing this tuning step—making context definition simple, composable, and portable across system instances—so aligned behavior can be reproduced without re-learning. Through the CERO loop (Configuration → Engagement → Recursion → Optimisation), friction is iteratively reduced over time rather than assumed to be zero.
One-liner: Digital 1 guesses; Digital 2 aligns; FlowZero makes alignment portable and iteratively reduces friction.
