## Profile/Group Creation, Staging, and Publishing (2026-04)


### New Features & Learnings (2026-04-23)
- The Index Edit window now guarantees that new group/profile files will never overwrite existing files:
	- When you click [Create Group], the app calls a backend API to list all existing `fzXXXX.json` files in the `public/` folder and picks the next available number, even if files were added outside the UI.
	- This ensures that all new group/profile, token, and sticky files are always created with a unique, incremental number.
	- The [Publish Group Files] button is also safe and will never overwrite an existing group file.
- The Add Entry and Download buttons have been removed to prevent accidental or incomplete additions.
- Only the [Create Group] and [Publish Group Files] buttons can add new groups, ensuring all associated files (profile, token, sticky) are created together.
- The profileId and fileName fields in the directory are now read-only to protect directory integrity; only description and extra fields are editable.
- A new [Save Changes] button allows you to persist edits to descriptions/comments in `fz_profile_index.json` by downloading/overwriting the file.
- All changes to the index are reflected in selectors and other UI components immediately after saving/publishing.
- UI improvements: clearer modal/box titles, increased spacing, and consistent button styling for a more user-friendly admin workbench.
- The workflow is now robust against accidental data loss or corruption and is ready for further user-facing UI development.


#### Example Group Creation & Publishing Workflow (Safe Numbering)
1. Click [Create Group] in Index Edit. The app checks all existing group/profile files in `public/` and always picks the next available number (e.g., `fz0003.json` if `fz0002.json` exists, even if files were added manually).
2. Three new files are created in `public/NewState/` with this unique number: profile, token, and sticky.
3. Edit the new group/profile as needed.
4. Click [Publish Group Files] to copy the staged files to `public/` (no overwrite risk).
5. Download and/or publish the updated `fz_profile_index.json` as needed.

---

### New Features & Learnings
- The text file editor (TextFileBackandSave.jsx) now supports editing and saving any `.txt` file by name.
- When saving, the filename is always taken from the React state (input field or file picker), so any valid `.txt` file can be created or updated in `public/NewState/`.
- When saving, if the file already exists, a backup is created with the `.bck` extension in the same folder.
- The refresh-to-public operation also supports any filename and creates a backup of the public file before overwriting (e.g., `public/text_2_test.txt.bck`).
- This means you can always roll back to the previous version by restoring the `.bck` file.
- The backend endpoints `/api/save-text` and `/api/refresh-text` now both accept a `filename` property in the POST body, defaulting to `text_text.txt` if not provided.
- This logic is robust for multi-file workflows and safe editing/publishing.

#### Example Workflow
1. Enter or pick a filename in the editor UI.
2. Edit and save: file is written to `public/NewState/` and backup is created if needed.
3. Refresh: file is copied to `public/` and backup is created if needed.
4. If a mistake is made, restore from the `.bck` file in either location.

# FlowZero Application Documentation (FZ_Appdocu)

## Overview
FlowZero is a zero-resistance workflow tool for managing and editing user configuration profiles (PUCKs), sticky notes, and personal tag tokens. The application is built with React and Vite, and operates on local JSON files for privacy and portability.

### Key Concepts
- **PUCK (Portable User Configuration Knowledge):** Local user profile JSON files, e.g., `fz0000.json`.
- **Sticky Notes:** Per-profile sticky note arrays, e.g., `fz0000_sticky.json`.
- **Tokens:** Per-profile tag tokens, e.g., `fz0000_token.json`.
- **User Index:** `fz_profile_index.json` and `fz_user_index.json` provide metadata and lookup for all profiles.



## Application UI and Capabilities

The FlowZero App UI is a modular, user-friendly admin workbench for managing all aspects of user configuration profiles (PUCKs), sticky notes, and tokens. The UI is organized around the core FlowZero loop: **Gestalt → Prune → PUCK → Prime → Prompt → Loop/Live**. Main features and components include:

- **Profile Edit:** Edit the array of fields for any selected profile (PUCK). Add, modify, or remove entries, and save changes as a new JSON file.
- **Token Edit:** Manage personal tag tokens (PTAGs) for the selected profile, including meanings and usage notes.
- **Sticky Edit:** Edit sticky notes (contextual notes, links, reminders) for each user profile.
- **Text File Edit:** Edit any `.txt` file in the system, with backup and safe publishing workflows (automatic backup before overwrite).
- **Index Edit:** View and edit the profile index (`fz_profile_index.json`).
	- **Create Puck:** The [Create Group] button safely creates a new PUCK (profile/group) with unique numbering, ensuring no overwrites. Associated token and sticky files are created together in `public/NewState/`.
	- **Publish Group Files:** The [Publish Group Files] button moves staged files to the public directory, always preventing overwrites.
- **Gestalt Viewer (Show Gestalt):** A modal window that displays a combined, read-only summary of the selected profile's main data (profile fields, tokens, sticky notes)—the full context field (Gestalt) before pruning to a PUCK. This supports the workflow: start with Gestalt, prune to relevant context, and package as a PUCK.


### Visibility Filtering (2026-04-25)
All Gestalt and PUCK creation workflows now support robust visibility filtering:
- **Visibility Selector:** In both the Gestalt Viewer and PUCK Creator, a dropdown lets you select between Secret (all data), Private (private+public), and Public (public only).
- **Filtered Views:** The Gestalt Viewer displays only the profile, token, and sticky records matching the selected visibility. The PUCK Creator allows you to select and export only the filtered set.
- **Export Safety:** When creating a PUCK, only the visible (filtered) profile, token, and sticky items are included in the export, ensuring that sensitive data is never accidentally included.
- **Consistent Logic:** The filtering logic is applied identically across all relevant UI components, and the codebase is fully linted and integrity-checked for reliability.

This enables a workflow where a user can:
1. View the full Gestalt (all context) filtered by visibility.
2. Prune and select only the desired items for a PUCK export.
3. Export a PUCK JSON for use as an AI prompt or calibration set, with confidence that only the intended data is included.

## Data Model Variation and Extensibility (2026-04-25)
FlowZero intentionally uses different field labels and structures for profiles, tokens, and stickies. This variation is by design:
- **Profiles** are structured, multi-field records for user or entity configuration.
- **Tokens** are compact, tag-like records for tagging, lookup, or quick reference.
- **Stickies** are lightweight, flexible modifiers that complement both profiles and tokens.

This separation allows each type to serve its unique purpose, keeps the system expressive, and supports future extensibility. The UI and data model are kept consistent, so the backend (file system or database) can be swapped out without changing the user experience.

> This documentation is written for programmers and AI agents. The goal is to enable anyone (human or AI) to generate a similar system or migrate to a new backend (e.g., a database) by following these principles and field label conventions. The separation of data structure and UI affordance is intentional and should be preserved in any implementation.

# FlowZero – Daily Specification (v0.1 Draft)

## 1. Overview

FlowZero is a lightweight, human-centric system for managing meaning, context, and action in everyday life. It enables individuals and small groups to tune their interaction with sytems using simple constructs that remain readable, flexible, and composable.

At its core, FlowZero solves a common problem:

> Humans operate in rich, contextual environments, but traditional systems force rigid structure or lose meaning entirely.

FlowZero provides a middle path:

* **natural language remains intact**
* **meaning is explicitly anchored**
* **context is dynamically applied**

This is achieved through three core primitives:

* **Profiles (context)**
* **Nomers (meaning anchors)**
* **Stickies (situational modifiers)**

Together, these form a minimal yet powerful system capable of handling a wide range of real-world coordination tasks, by using portable "Modes" that fine tune receiving systems.

---

## 2. Core Concepts

### 2.1 Profiles

A **Profile** represents any identifiable entity:

* Person (Chris, Pia, Uli)
* Place (Dachau, Allach, Practice Room)
* Object (Mitch guitar, generator, car)
* Environment (restaurant, building, gig setup)

Profiles serve as containers for:

* Nomers
* JAR definitions
* Norms (rules and conventions)
* Contextual knowledge

Profiles define:

> **Where and for whom meaning applies**

---

### 2.2 Nomers

A **Nomer** is a labeled handle that represents a concept with a defined meaning.

Nomers can be:

* existing words with specialized meaning (“garlic”)
* nicknames (“Mitch”)
* acronyms (“EMC”)
* invented terms (“foodeldunk”)
* phrases (“the usual”)

A nomer is not defined by its form, but by its function:

> A nomer is a **locally fixed meaning for a handle**

Nomers are written in square brackets:

```text
[Mitch]
[Dachau]
[Breakfast]
```

---

### 2.3 Tokens

A **Token** is the system-level representation of a nomer.

* Always enclosed in square brackets
* Used for parsing and recognition

```text
[Mitch]
```

A token represents a nomer but is not the meaning itself.

---

### 2.4 JAR (JSON Accessible Render)

A **JAR** is the structured meaning package behind a nomer.

It can contain:

* definition
* properties
* relationships to other nomers
* media references (images, links)
* usage notes and quirks

Example:

```json
{
  "Mitch": {
    "type": "guitar",
    "description": "Fender Telecaster, Hurricane Mitch mahogany",
    "tools": ["2.5mm hex", "long trem arm"],
    "notes": "slightly sensitive tuning, expressive tone"
  }
}
```

Conceptually:

* label = outside of the jar
* JAR = everything inside

JARs are:

* structured
* portable
* expandable on demand

---

### 2.5 Stickies

**Stickies** represent situational context.

They:

* modify nomers or profiles
* override defaults
* are temporary and dynamic

Example:

```text
[Breakfast] (for 3, at [Dachau], skip eggs)
```

Stickies allow the system to express:

> “What is different today”

* system-recognized meaning

```

Indicates:

* nomer recognized

## Profile/Group Creation, Staging, and Publishing (2026-04)


### New Features & Learnings (2026-04-23)
- The Index Edit window now guarantees that new group/profile files will never overwrite existing files:
	- When you click [Create Group], the app calls a backend API to list all existing `fzXXXX.json` files in the `public/` folder and picks the next available number, even if files were added outside the UI.
	- This ensures that all new group/profile, token, and sticky files are always created with a unique, incremental number.
	- The [Publish Group Files] button is also safe and will never overwrite an existing group file.
- The Add Entry and Download buttons have been removed to prevent accidental or incomplete additions.
- Only the [Create Group] and [Publish Group Files] buttons can add new groups, ensuring all associated files (profile, token, sticky) are created together.
- The profileId and fileName fields in the directory are now read-only to protect directory integrity; only description and extra fields are editable.
- A new [Save Changes] button allows you to persist edits to descriptions/comments in `fz_profile_index.json` by downloading/overwriting the file.
- All changes to the index are reflected in selectors and other UI components immediately after saving/publishing.
- UI improvements: clearer modal/box titles, increased spacing, and consistent button styling for a more user-friendly admin workbench.
- The workflow is now robust against accidental data loss or corruption and is ready for further user-facing UI development.


#### Example Group Creation & Publishing Workflow (Safe Numbering)
1. Click [Create Group] in Index Edit. The app checks all existing group/profile files in `public/` and always picks the next available number (e.g., `fz0003.json` if `fz0002.json` exists, even if files were added manually).
2. Three new files are created in `public/NewState/` with this unique number: profile, token, and sticky.
3. Edit the new group/profile as needed.
4. Click [Publish Group Files] to copy the staged files to `public/` (no overwrite risk).
5. Download and/or publish the updated `fz_profile_index.json` as needed.

---

### New Features & Learnings
- The text file editor (TextFileBackandSave.jsx) now supports editing and saving any `.txt` file by name.
- When saving, the filename is always taken from the React state (input field or file picker), so any valid `.txt` file can be created or updated in `public/NewState/`.
- When saving, if the file already exists, a backup is created with the `.bck` extension in the same folder.
- The refresh-to-public operation also supports any filename and creates a backup of the public file before overwriting (e.g., `public/text_2_test.txt.bck`).
- This means you can always roll back to the previous version by restoring the `.bck` file.
- The backend endpoints `/api/save-text` and `/api/refresh-text` now both accept a `filename` property in the POST body, defaulting to `text_text.txt` if not provided.
- This logic is robust for multi-file workflows and safe editing/publishing.

#### Example Workflow
1. Enter or pick a filename in the editor UI.
2. Edit and save: file is written to `public/NewState/` and backup is created if needed.
3. Refresh: file is copied to `public/` and backup is created if needed.
4. If a mistake is made, restore from the `.bck` file in either location.

# FlowZero Application Documentation (FZ_Appdocu)

## Overview
FlowZero is a zero-resistance workflow tool for managing and editing user configuration profiles (PUCKs), sticky notes, and personal tag tokens. The application is built with React and Vite, and operates on local JSON files for privacy and portability.

### Key Concepts
- **PUCK (Portable User Configuration Knowledge):** Local user profile JSON files, e.g., `fz0000.json`.
- **Sticky Notes:** Per-profile sticky note arrays, e.g., `fz0000_sticky.json`.
- **Tokens:** Per-profile tag tokens, e.g., `fz0000_token.json`.
- **User Index:** `fz_profile_index.json` and `fz_user_index.json` provide metadata and lookup for all profiles.



## Application UI and Capabilities

The FlowZero App UI is a modular, user-friendly admin workbench for managing all aspects of user configuration profiles (PUCKs), sticky notes, and tokens. The UI is organized around the core FlowZero loop: **Gestalt → Prune → PUCK → Prime → Prompt → Loop/Live**. Main features and components include:

- **Profile Edit:** Edit the array of fields for any selected profile (PUCK). Add, modify, or remove entries, and save changes as a new JSON file.
- **Token Edit:** Manage personal tag tokens (PTAGs) for the selected profile, including meanings and usage notes.
- **Sticky Edit:** Edit sticky notes (contextual notes, links, reminders) for each user profile.
- **Text File Edit:** Edit any `.txt` file in the system, with backup and safe publishing workflows (automatic backup before overwrite).
- **Index Edit:** View and edit the profile index (`fz_profile_index.json`).
	- **Create Puck:** The [Create Group] button safely creates a new PUCK (profile/group) with unique numbering, ensuring no overwrites. Associated token and sticky files are created together in `public/NewState/`.
	- **Publish Group Files:** The [Publish Group Files] button moves staged files to the public directory, always preventing overwrites.
- **Gestalt Viewer (Show Gestalt):** A modal window that displays a combined, read-only summary of the selected profile's main data (profile fields, tokens, sticky notes)—the full context field (Gestalt) before pruning to a PUCK. This supports the workflow: start with Gestalt, prune to relevant context, and package as a PUCK.


### Visibility Filtering (2026-04-25)
All Gestalt and PUCK creation workflows now support robust visibility filtering:
- **Visibility Selector:** In both the Gestalt Viewer and PUCK Creator, a dropdown lets you select between Secret (all data), Private (private+public), and Public (public only).
- **Filtered Views:** The Gestalt Viewer displays only the profile, token, and sticky records matching the selected visibility. The PUCK Creator allows you to select and export only the filtered set.
- **Export Safety:** When creating a PUCK, only the visible (filtered) profile, token, and sticky items are included in the export, ensuring that sensitive data is never accidentally included.
- **Consistent Logic:** The filtering logic is applied identically across all relevant UI components, and the codebase is fully linted and integrity-checked for reliability.

This enables a workflow where a user can:
1. View the full Gestalt (all context) filtered by visibility.
2. Prune and select only the desired items for a PUCK export.
3. Export a PUCK JSON for use as an AI prompt or calibration set, with confidence that only the intended data is included.

### TokenEdit.jsx
- Loads user index and allows selection of a profile's token file.
- Edits array of tokens (fields: Token, Meaning, Usage).
- No manual file open; always associated with a profile.

- Simplicity for users and developers
- Consistent UI and data model

## Data Model Variation and Extensibility (2026-04-25)
FlowZero intentionally uses different field labels and structures for profiles, tokens, and stickies. This variation is by design:
- **Profiles** are structured, multi-field records for user or entity configuration.
- **Tokens** are compact, tag-like records for tagging, lookup, or quick reference.
- **Stickies** are lightweight, flexible modifiers that complement both profiles and tokens.

This separation allows each type to serve its unique purpose, keeps the system expressive, and supports future extensibility. The UI and data model are kept consistent, so the backend (file system or database) can be swapped out without changing the user experience.

> This documentation is written for programmers and AI agents. The goal is to enable anyone (human or AI) to generate a similar system or migrate to a new backend (e.g., a database) by following these principles and field label conventions. The separation of data structure and UI affordance is intentional and should be preserved in any implementation.

# FlowZero – Daily Specification (v0.1 Draft)

## 1. Overview

FlowZero is a lightweight, human-centric system for managing meaning, context, and action in everyday life. It enables individuals and small groups to tune their interaction with sytems using simple constructs that remain readable, flexible, and composable.

At its core, FlowZero solves a common problem:

> Humans operate in rich, contextual environments, but traditional systems force rigid structure or lose meaning entirely.

FlowZero provides a middle path:

* **natural language remains intact**
* **meaning is explicitly anchored**
* **context is dynamically applied**

This is achieved through three core primitives:

* **Profiles (context)**
* **Nomers (meaning anchors)**
* **Stickies (situational modifiers)**

Together, these form a minimal yet powerful system capable of handling a wide range of real-world coordination tasks, by using portable "Modes" that fine tune receiving systems.

---

## 2. Core Concepts

### 2.1 Profiles

A **Profile** represents any identifiable entity:

* Person (Chris, Pia, Uli)
* Place (Dachau, Allach, Practice Room)
* Object (Mitch guitar, generator, car)
* Environment (restaurant, building, gig setup)

Profiles serve as containers for:

* Nomers
* JAR definitions
* Norms (rules and conventions)
* Contextual knowledge

Profiles define:

> **Where and for whom meaning applies**

---

### 2.2 Nomers

A **Nomer** is a labeled handle that represents a concept with a defined meaning.

Nomers can be:

* existing words with specialized meaning (“garlic”)
* nicknames (“Mitch”)
* acronyms (“EMC”)
* invented terms (“foodeldunk”)
* phrases (“the usual”)

A nomer is not defined by its form, but by its function:

> A nomer is a **locally fixed meaning for a handle**

Nomers are written in square brackets:

```text
[Mitch]
[Dachau]
[Breakfast]
```

---

### 2.3 Tokens

A **Token** is the system-level representation of a nomer.

* Always enclosed in square brackets
* Used for parsing and recognition

```text
[Mitch]
```

A token represents a nomer but is not the meaning itself.

---

### 2.4 JAR (JSON Accessible Render)

A **JAR** is the structured meaning package behind a nomer.

It can contain:

* definition
* properties
* relationships to other nomers
* media references (images, links)
* usage notes and quirks

Example:

```json
{

# FlowZero – Daily Specification (v0.1 Draft)

## 1. Overview

FlowZero is a lightweight, human-centric system for managing meaning, context, and action in everyday life. It enables individuals and small groups to tune their interaction with sytems using simple constructs that remain readable, flexible, and composable.

At its core, FlowZero solves a common problem:

> Humans operate in rich, contextual environments, but traditional systems force rigid structure or lose meaning entirely.

FlowZero provides a middle path:

* **natural language remains intact**
* **meaning is explicitly anchored**
* **context is dynamically applied**

This is achieved through three core primitives:

* **Profiles (context)**
* **Nomers (meaning anchors)**
* **Stickies (situational modifiers)**

Together, these form a minimal yet powerful system capable of handling a wide range of real-world coordination tasks, by using portable "Modes" that fine tune receiving systems.

## 2. Core Concepts

### 2.1 Profiles

A **Profile** represents any identifiable entity:

* Person (Chris, Pia, Uli)
* Place (Dachau, Allach, Practice Room)
* Object (Mitch guitar, generator, car)
* Environment (restaurant, building, gig setup)

Profiles serve as containers for:

* Nomers
* JAR definitions
* Norms (rules and conventions)
* Contextual knowledge

Profiles define:

> **Where and for whom meaning applies**

---

### 2.2 Nomers

A **Nomer** is a labeled handle that represents a concept with a defined meaning.

Nomers can be:

* existing words with specialized meaning (“garlic”)
* nicknames (“Mitch”)
* acronyms (“EMC”)
* invented terms (“foodeldunk”)
* phrases (“the usual”)

A nomer is not defined by its form, but by its function:

> A nomer is a **locally fixed meaning for a handle**

Nomers are written in square brackets:

```text
[Mitch]
[Dachau]
[Breakfast]
```

### 2.3 Tokens

A **Token** is the system-level representation of a nomer.

* Always enclosed in square brackets
* Used for parsing and recognition

```text

```

A token represents a nomer but is not the meaning itself.

---

A **JAR** is the structured meaning package behind a nomer.

It can contain:

* definition
* properties
* relationships to other nomers

[Gamer's Fallacy]: The Gamer's Fallacy (GF) is a type of Paradigm Frame Lock (PFL) where expectations shaped by highly responsive, configurable environments are unconsciously applied to the real world. These environments can include interactive gaming systems (where outcomes are controllable and failures are reversible) as well as pampered or high-service contexts (e.g. service staff, concierge, or butler-style support) where individual needs are consistently anticipated and fulfilled. Under GF, people may expect high configurability, low-cost failure, and that systems or others will adapt to their personal preferences. In real-world contexts—governed by shared constraints, policies, and non-negotiable trade-offs—this leads to friction. In edge cases, GF can manifest as entitled behavior, where individuals expect others to adjust to their needs, quirks, and idiosyncrasies without reciprocal adaptation.
One-liner: Gamer’s Fallacy = expecting real-world systems to behave like configurable, low-risk environments.

===== FZ_How2UsePuck4LLM.txt ===== 
===============================
PATCH: PUCK LOAD PROTOCOL v0.1
===============================

PURPOSE
-------
Ensure the LLM correctly interprets the PUCK before using it.
Make sure the content of previously loaded PUCKs is refreshed by the new upload.

===============================
PATCH: PUCK INGEST & TOKEN ECHO v0.1
===============================

PURPOSE
-------
Ensure that uploaded PUCK content is not only received but actively parsed and made visible.

MANDATORY INGEST STEP
---------------------
After uploading the PUCK archive (puck_*.zip):

1. Unpack the archive
2. Read all fz_*.* files
3. Extract token definitions from all *_token.json files

TOKEN ECHO (REQUIRED)
---------------------
Display a concise list of active tokens and their meanings.

Example:
[Mitch] → Fender Telecaster
[Sarah] → strict vegan
[Fiona] → husband Ross

This confirms that:
- PUCK is accessible
- tokens are correctly parsed
- latest version is active

FAILURE MODE
------------
If tokens are not displayed:
→ PUCK is not properly ingested
→ DO NOT proceed with reasoning

PRINCIPLE
---------
Visibility before trust.

ONE-LINER
---------
“If you can’t show the tokens, you can’t use the PUCK.”

===============================
END PATCH
===============================

LOAD ORDER (MANDATORY)
----------------------
1. Upload and process this instruction file (How2UsePuck4LLM.txt)
2. Upload PUCK archive (puck_*.zip containing fz_*.* files)
3. Display acknowledgement of PUCK upload and date.

The instruction file defines how to interpret tokens and mappings.
The PUCK provides the actual content.

Do not reverse the order.

VALIDATION STEP (REQUIRED)
--------------------------
This PUCK (Personal Usage Configuration Knowledge) encodes a user’s microverse: personal tokens, preferences, routines, and shorthand mappings. Tokens (e.g. [Mitch], [Skinny Thing]) are compressed references that expand into specific objects, people, places, or actions. Interpret all bracketed terms via token files before reasoning. Do not assume generic meaning if a token exists. Combine tokens with context (time, routines like “usual Wednesday”) to reconstruct intent. Prefer expansion over guessing. Preserve user-specific semantics over global defaults. If ambiguity remains, ask for clarification or suggest likely expansions. Goal: align with the user’s language so communication is low-friction, accurate, and immediately actionable (“click”).
