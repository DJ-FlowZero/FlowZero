### New Features & Learnings
- The text file editor (TextFileBackandSave.jsx) now supports editing and saving any `.txt` file by name.
- When saving, the filename is always taken from the React state (input field or file picker), so any valid `.txt` file can be created or updated in `public/NewState/`.
- When saving, if the file already exists, a backup is created with the `.bck` extension in the same folder.
- The refresh-to-public operation also supports any filename and creates a backup of the public file before overwriting (e.g., `public/text_2_test.txt.bck`).
- This means you can always roll back to the previous version by restoring the `.bck` file.
- The backend endpoints `/api/save-text` and `/api/refresh-text` now both accept a `filename` property in the POST body, defaulting to `text_text.txt` if not provided.
- This logic is robust for multi-file workflows and safe editing/publishing.

#### Example Workflow
1. Enter or pick a filename in the editor UI.
2. Edit and save: file is written to `public/NewState/` and backup is created if needed.
3. Refresh: file is copied to `public/` and backup is created if needed.
4. If a mistake is made, restore from the `.bck` file in either location.

# Application Startup & Configuration (2026-04)

## Quick Start

1. **Install dependencies:**
	 - Open a terminal in the project root and run: `npm install`
2. **Start the app (backend + frontend):**
	 - Run: `npm run dev`
	 - This launches both the backend (Express, on http://localhost:3001) and the frontend (Vite/React, on http://localhost:5173) together.
	 - The app will be available at [http://localhost:5173/](http://localhost:5173/)
	 - **Session shortcut:** `start meup` in terminal runs `meup.bat`, which wraps `npm run dev`. Standard session startup command.

## Scripts (package.json)

```
"dev:backend": "nodemon backend.cjs"      # Starts the backend API server
"dev:frontend": "vite"                     # Starts the Vite React frontend
"dev": "concurrently \"npm run dev:backend\" \"npm run dev:frontend\""  # Runs both together
```

## Configuration

- **Backend:**
	- Main file: `backend.cjs` (Express server, all file persistence)
	- Port: 3001
	- Serves and manages files in `public/`, `public/Gestalt/`, `public/fz_txt/`, `public/fz_md/`, `public/fz_PUCK_data/`
  - Key endpoints: `/api/save-profile`, `/api/txt/load`, `/api/txt/save`, `/api/txt/restore`, `/api/list-txt-files`, `/api/list-md-files`, `/api/save-puck`, `/api/save-puck-index`, `/api/profile/backups`, `/api/profile/restore`, `/api/health`, `/api/list-fz-files`
	- Backup pattern: timestamped .bck.YYYY-MM-DDTHH-MM-SS in backups/ subdirectory next to source file; last 10 kept per file
- **Frontend:**
	- React app (src/), served by Vite
	- Port: 5173
	- Proxy: API requests to `/api` are proxied to the backend (see `vite.config.js`)

## Customization

- To change ports, edit the `PORT` in `backend.cjs` and the `server` section in `vite.config.js`.
- To add new API endpoints, edit `backend.cjs`.
- To add new UI features, edit React components in `src/`.

# FlowZero Application Documentation (FZ_Appdocu)

## Overview
FlowZero is a zero-resistance workflow tool for managing and editing user configuration profiles (PUCKs), sticky notes, and personal tag tokens. The application is built with React and Vite, and operates on local JSON files for privacy and portability.

### Key Concepts
- **FlowSet:** The stored unit of context in FlowZero. Three linked files for a single entity (person, place, object, AI, environment): Profile (fzXXXX.json) + Flexicon (fzXXXX_token.json) + Sticky (fzXXXX_sticky.json). A FlowSet is the source — a Gestalt is what you get when you compile it.
- **Gestalt:** The compiled output of a FlowSet at a given visibility level (Public/Private/Secret). The unfiltered, unpruned context field before any PUCK export. Viewed via Show Gestalt modal.
- **PUCK (Portable Use Case Knowledge):** Pruned, compiled, portable constraint surface exported from a Gestalt. NOT a raw data dump — a task-relevant row selection, optionally including Flow Calibration. Export file: e.g. `PUCK_export_fz0000.txt`.
- **Job Package (CompositePUCK):** A job brief + one or more PUCKs combined into a single priming document. Generated by Compose Job Package. Ephemeral — created per task, not stored.
- **BioLogic:** The explicit, portable operating system beneath prompts — how a person reasons, what they tolerate, what needs stating vs. implying. Made machine-readable via Flow Calibration parameters. AI-agnostic: readable by any model, no fine-tuning required.
- **Flow Calibration:** 18 behavioral parameters (response length, tone, challenge level, etc.) stored in `FZ_Flow_Config.json`. Edited via the Calibrate Flow panel. Optionally included in PUCK export as a plain-text block.
- **Profile:** The stable-facts member of a FlowSet. Label/value pairs: name, constraints, preferences, background. File: `fzXXXX.json`.
- **Flexicon:** The meaning-handle member of a FlowSet. Personal CCML dictionary — the user's set of [token] :: definitions in their own vocabulary. File: `fzXXXX_token.json`. (Was called Token/Token Edit.)
- **Sticky:** The runtime-modifier member of a FlowSet. What is different right now: situation, active incidents, flags. File: `fzXXXX_sticky.json`.
- **CCML (Context Configuration Markup Language):** 7-primitive markup for encoding human context — `[token]`, `{type}`, `::`, `<flag>`, `//note`, `@trigger`, `#tag`. Human-writable, AI-parseable, language-neutral. Core is hard-capped at seven primitives; extensions go to XCML. Full spec: FZ_06_CCML_Spec.txt.
- **ServCon (Service Contract):** Session-level behavioral agreement between HUM and AI. Defined in FZ_01_AI_ServCon.txt. Upload at session start. AI alias: Loop. Protocol: [LOOP/HUM].



## Application UI and Capabilities

The FlowZero App UI is a modular admin workbench organized around the core chain:
**FlowSet → Gestalt → Prune → PUCK → Job Package → Prime AI**

Main features and components:

- **FlowSet Definition:** Create a FlowSet. [Create FlowSet] writes all three member files to Gestalt/ immediately and updates the index. ? opens help.
- **Profile Editor:** Edit the stable-facts array for any selected FlowSet. Label/value/flag/todo entries.
- **Flexicon Editor:** Manage the meaning-handle tokens for the selected FlowSet. Token/Meaning/Usage/flag entries. (Was Token Edit.)
- **Sticky Edit:** Edit runtime modifiers for each FlowSet. Stickytype/note/links/flag entries.
- **Text File Editor:** Edit any `.txt` file in Gestalt/ or fz_txt/. Dropdown populated live from `/api/list-txt-files`. Automatic timestamped backup on every save. Version history panel shows last 10 backups with Restore.
- **MD Editor:** Edit any `.md` file in fz_md/ (or fz_txt/). Raw/Preview toggle — Preview renders markdown inline. Same chrome as Text File Editor (unsaved pill, Reload, Save, Version History). Populated from `/api/list-md-files`.
- **Help System:** Two modes — (1) HelpModal overlay triggered by ? HelpBtn buttons inline with each toolbar button when Onscreen Help is On. Loads any fz_md/ file and renders as a formatted doc. (2) Application Overview inline panel below the top bar (toggleable pill button). All help content is editable in MD Editor without code changes.
- **Calibrate Flow:** Edit the 18 BioLogic behavioral parameters stored in `FZ_Flow_Config.json`. Pill-button radio UI per parameter. Free-text notes area. Save to Gestalt or Download JSON. Parameters include: response length, tone register, challenge level, ambiguity handling, scope discipline, and 13 others.

### Visibility Filtering (2026-04-25)
All Gestalt and PUCK creation workflows now support robust visibility filtering:
- **Visibility Selector:** In both the Gestalt Viewer and PUCK Creator, a dropdown lets you select between Secret (all data), Private (private+public), and Public (public only).
- **Filtered Views:** The Gestalt Viewer displays only the profile, token, and sticky records matching the selected visibility. The PUCK Creator allows you to select and export only the filtered set.
- **Export Safety:** When creating a PUCK, only the visible (filtered) profile, token, and sticky items are included in the export, ensuring that sensitive data is never accidentally included.
- **Consistent Logic:** The filtering logic is applied identically across all relevant UI components, and the codebase is fully linted and integrity-checked for reliability.

This enables a workflow where a user can:
1. View the full Gestalt (all context) filtered by visibility.
2. Prune and select only the desired items for a PUCK export.
3. Export a PUCK JSON for use as an AI prompt or calibration set, with confidence that only the intended data is included.

## Main Components
All components follow the same pattern: load profile index via `fetchUserIndex()`, allow profile selection from dropdown, load/edit the corresponding JSON file.

Data shapes per editor:
- **Jedit:** `{ profile: [ { label, value, flag, todo } ] }`
- **StickyEdit:** `{ sticky: [ { stickytype, note, links, flag } ] }` — reserved stickytype values: `"gap"` (unverified info), `"fault"` (active incident)
- **TokenEdit:** `{ FZ_Tokens: [ { Token, Meaning, Usage, flag } ] }`
- **CalibrateFlow:** `{ params: [ { id, label, description, options[3], value } ] }` — 18 params in FZ_Flow_Config.json

Path rule: never construct Gestalt paths inline. Always use `getGestaltFileUrl(filename)` from `userIndexUtil.js`.

## Data Model
- Each profile file is a JSON object with a single key (profileId or similar) mapping to an array of entries.
- Sticky and token files follow the same pattern, with different field names.

3. User can edit, add, or remove entries.

## Backend API Contract

All endpoints are served by backend.cjs on port 3001.
All POST bodies are JSON (Content-Type: application/json).
All responses are JSON.
Path traversal is prevented server-side via safeFilePath() — filenames with directory
separators are rejected with 400.

---

### POST /api/save-profile
Writes a profile, token, or sticky JSON file to the Gestalt directory. Creates a
timestamped backup in Gestalt/backups/ before overwrite. Prunes to last 10 backups.

Request body:
  { "filename": "fz0001.json", "content": "<JSON string>" }

  filename  required  Target filename (no path separators). Must exist within Gestalt dir.
  content   required  Full file content as a string (already serialized JSON).

Success response (200):
  { "status": "ok", "message": "Saved at YYYY-MM-DD HH:MM:SS", "backups": ["fz0001.json.bck.2026-05-01T12-00-00", ...] }

Error responses:
  400  { "status": "error", "message": "Missing filename." }
  400  { "status": "error", "message": "Invalid filename." }
  500  { "status": "error", "message": "Failed to save profile." }

---

### GET /api/profile/backups
Returns the list of timestamped backups for a given file in Gestalt/backups/.

Query params:
  filename  required  Bare filename (e.g. fz0001.json)

Success response (200):
  { "status": "ok", "backups": ["fz0001.json.bck.2026-05-01T12-00-00", ...] }

---

### GET /api/profile/restore
Reads a backup file content from Gestalt/backups/.

Query params:
  filename  required  Bare source filename (e.g. fz0001.json)
  file      required  Backup filename (e.g. fz0001.json.bck.2026-05-01T12-00-00)

Success response (200):
  { "status": "ok", "content": "<file text>" }

Error responses:
  400  { "status": "error", "message": "Invalid parameters." }
  404  { "status": "error", "message": "Backup not found." }

---

### GET /api/txt/load
Loads a text or markdown file for editing. Accepts fetchPath in dir/filename format.
Supported dirs: fz_txt, Gestalt, fz_md.

Query params:
  fetchPath  required  e.g. "fz_txt/FZ_99_DailyBlog.txt" or "fz_md/FZ_10_ProductPitch.md"

Success response (200):
  { "status": "ok", "content": "<file text>", "backups": ["filename.bck.2026-05-01T12-00-00", ...] }

Error responses:
  400  { "status": "error", "message": "Invalid path." }
  404  { "status": "error", "message": "File not found." }

### POST /api/save-puck
Writes a PUCK .txt file to public/fz_PUCK_data/ and upserts fz_puck_index.json.
Called by PuckCreator.handleExport immediately before the browser download.

Request body:
  {
    "filename":   "PUCK_export_fz0001.json.txt",
    "content":    "<full PUCK text>",
    "indexEntry": { "name": "...", "filename": "...", "sourceProfiles": "...",
                    "visibility": "...", "date": "YYYY-MM-DD", "description": "" }

  filename    required  Bare filename; path separators rejected (safeFilePath guard).

  400  { "status": "error", "message": "Missing filename or content." }

## Error Handling Policy

Frontend components follow this contract for all API calls:

1. Non-OK HTTP status (4xx/5xx): display the `message` field from the response JSON
   as a user-visible status string. Fall back to the HTTP status text if no body.
2. Network failure (fetch throws): display a generic "Connection error — is the backend
   running?" message. Do not swallow silently.
3. Backup failures (logged server-side): do not surface to the user unless the primary
   write also fails.
4. Missing file on read: display "File not found" status. Do not crash the component.
5. All status messages are transient — they disappear on the next user action.
   No persistent error state in components.

Backend follows this contract:
- 400 for missing or invalid request parameters.
- 404 for missing target files.
- 500 for unexpected I/O errors.
- All errors return { "status": "error", "message": "..." }.
- Backup failures are logged to console but do not fail the primary operation.

## UI String Externalization — App_UI_ext.json (2026-05-01)

All toolbar button labels and visible UI header strings are stored in `public/App_UI_ext.json`.

Key pattern: each entry is `{ "value": "Display Text", "flag": "public" }`.
Key naming: `fz_btn_<identifier>` for all toolbar buttons. E.g. `fz_btn_profile_edit`.

Loading pattern in App.jsx:
1. `uiStrings` state initialized to `null`.
2. `useEffect` on mount: `fetch('/App_UI_ext.json')` → sets `uiStrings`.
3. `uiStr(key)` helper: if `uiStrings` null → returns `'…'` (loading). If key missing/empty → returns `'⚠ Load Error'`. Otherwise returns `value`.
4. All `<FzButton>` labels in the toolbar call `uiStr('fz_btn_*')`.

Current fz_btn_* keys (2026-05-01):
- `fz_btn_flowset_index`  → "FlowSet Definition"
- `fz_btn_profile_edit`   → "Profile Editor"
- `fz_btn_flexicon_edit`  → "Flexicon Editor"
- `fz_btn_sticky_edit`    → "Sticky Editor"

## Field Labels Policy (2026-04-25)
To keep the FlowZero system lean, maintainable, and user-friendly, the number of field labels (column headers) in all editors is intentionally limited. This ensures:
- Simplicity for users and developers
- Consistent UI and data model
- Easier future maintenance and extension

If additional fields are needed, they can be added through configuration or future updates, but the default set is kept minimal by design.

---

## Data Model Variation and Extensibility (2026-04-25)
FlowZero intentionally uses different field labels and structures for profiles, tokens, and stickies. This variation is by design:
- **Profiles** are structured, multi-field records for user or entity configuration.
- **Tokens** are compact, tag-like records for tagging, lookup, or quick reference.
- **Stickies** are lightweight, flexible modifiers that complement both profiles and tokens.

This separation allows each type to serve its unique purpose, keeps the system expressive, and supports future extensibility. The UI and data model are kept consistent, so the backend (file system or database) can be swapped out without changing the user experience.

> This documentation is written for programmers and AI agents. The goal is to enable anyone (human or AI) to generate a similar system or migrate to a new backend (e.g., a database) by following these principles and field label conventions. The separation of data structure and UI affordance is intentional and should be preserved in any implementation.

## Munchhausen Architecture Principle (MAP) — LOCKED

Design goal: "Compress definitions until reconstruction becomes trivial."

A cold reader (human or AI) with zero prior context must be able to reconstruct
the system from only the fz_txt/ artifact set. If reconstruction requires inference,
the artifact is underspecified.

Rules:
- Every non-obvious architectural decision must have a trace in fz_txt/ or config.json.
- fz_txt/ is the MAP artifact store — not code comments, not README.
- Reconstruction requires assembly only — no prior context, no inference.

This is an optimization target, not a soft aspiration. All fz_txt/ files are held to this standard.
When reviewing or updating any fz_txt/ file, ask: can someone with no history rebuild this from only these files?
If no → compress and specify further.

# Profile Editor

> The **Profile** holds the stable facts about a person, place, or entity — things that rarely change.

Each entry is a label/value pair with a visibility flag. Think of it as a structured background briefing — name, constraints, preferences, background.

---

## Fields

| Field | Purpose |
|-------|---------|
| **Label** | What the fact is — e.g. "Dietary constraint", "Location" |
| **Value** | The fact itself |
| **Flag** | Visibility — `public`, `private`, or `secret` |
| **Todo** | Optional follow-up note for incomplete or unverified entries |

## When to use

Fill in Profile first — it is the stable foundation. Stickies override it at runtime. The Flexicon names the things referenced inside it.

---

### FILE: fz_md/FZ_help_FlexiconEdit.md

# Flexicon Editor

> The **Flexicon** is a personal vocabulary — a dictionary of named things that matter to this entity, written in their own words.

Each token is a locally-fixed meaning handle: a name mapped to a structured definition. The token preserves the entity's exact language — no normalisation at capture time.

---

## Fields

| Field | Purpose |
|-------|---------|
| **Token** | The name in the entity's own words — `[Schlappen]`, `[Leo206]`, `[Wednesday]` |
| **Meaning** | What it resolves to — free text, any language |
| **Usage** | Where or when this token appears |
| **Flag** | Visibility — `public`, `private`, or `secret` |

## Why it matters

The Flexicon turns personal shorthand into AI-parseable context. A carer querying in Polish resolves `[Schlappen]` through the token, not the original word. Cross-language semantic bridge — built by one person, readable by any.

### FILE: fz_md/FZ_help_StickyEdit.md

# Sticky Editor

> **Stickies** are runtime modifiers — what is different *right now* about this entity.

Use stickies for anything situational, temporary, or currently active. They supplement or override the Profile for the current task without altering the stable base.

---

## Fields

| Field | Purpose |
|-------|---------|
| **Type** | Category — `note`, `gap`, `fault`, or any custom label |
| **Note** | The situational content |
| **Links** | Optional references — URLs, file paths |
| **Flag** | Visibility — `public`, `private`, or `secret` |

## Reserved types

| Type | Meaning |
|------|---------|
| `gap` | Information that is missing or unverified — flag for follow-up |
| `fault` | Active incident or known problem — visible in every PUCK until resolved |
