
## [2026-04-27] Directory Path Management

- All code now references the Gestalt directory via the FZ_GPATH variable in config.json.
- Changing the Gestalt directory only requires updating config.json and restarting servers.
- This pattern is recommended for all future directory or file path dependencies.

Data Model Overview
------------------
- fz_profile_index.json: Array of profile/group index entries (realName, profileId, fileName, description, type, tag, etc.)
- fzXXXX.json: { profile: [ { label, value, flag, todo, ... } ] }
- fzXXXX_token.json: { FZ_Tokens: [ { Token, Meaning, Usage, flag, ... } ] }
- fzXXXX_sticky.json: { sticky: [ { stickytype, note, links, flag, ... } ] }

Key Workflows
-------------
1. Profile/Group Creation: Use IndexEdit to create a new group/profile, which generates new fzXXXX.json, *_token.json, *_sticky.json in NewState/.
2. Editing: Use Jedit, TokenEdit, StickyEdit, or TextFileBackandSave to modify data.
3. Publishing: Use IndexEdit to publish new/edited files from NewState/ to Gestalt/ and update fz_profile_index.json.
4. Export: Use PuckCreator to export a PUCK (profile+token+sticky) as a .txt file with loader readme.
5. Backup/Restore: All saves create .bck files; Recyclebin/ holds old versions for manual recovery.

MAP Munchhausen Architecture Principle (Onboarding & Recovery)
-------------------------------------------------------------

Onboarding a New Developer
- Prerequisites: Node.js, npm, VS Code (recommended), Git
- Clone repo, run `npm install`, then `npm run dev` to start backend/frontend
- See FZ_Appdocu.txt and this file for architecture, code plan, and workflows

Recovery/Restore Instructions
- To restore any .json or .txt file, copy the corresponding .bck file (in NewState/, Gestalt/, or fz_txt/) to the original filename
- To recover deleted/overwritten profiles, copy from Recyclebin/ to Gestalt/
- For full reset, re-clone repo and copy in latest Gestalt/ and fz_txt/ files
