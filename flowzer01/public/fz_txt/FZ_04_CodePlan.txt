
# [2026-04-30] PROTOCOL — Strategic Architecture Frame

PROTOCOL is a retrospective backronym describing FlowZero's emergent identity as a
protocol layer for human context — not just a profile management tool.

  Portable        context travels with the person, not trapped in platforms
  Reliable        transmitted context survives transfer without distortion
  Open            no dependency on single vendor, model, or ecosystem
  Transactional   most useful interactions are bounded exchanges with clear intent
  Operational     context exists to enable action, not archival storage
  Context         context is the actual payload — not UI, not features
  Orchestration   intelligence (human, AI, robotic, procedural) coordinated against local reality
  Layer           FlowZero sits above Digital Air, tuning generic intelligence into local familiarity

Long-term aspiration: a common contextual transport layer between humans, AI systems,
services, and robots — analogous to TCP/IP for networks.

This frame emerged from Demo V1.0. It was not the starting vision in explicit form.
In retrospect, it may have been the shape of it all along.

# Test Scope

No formal test framework is specified for V1. Minimal required validation per feature:

| Feature                  | Validation method                                              |
|--------------------------|----------------------------------------------------------------|
| Profile/Token/Sticky edit | Manual: edit a field, save, reload, confirm value persisted    |
| Backup on save           | Manual: confirm .bck file created in Gestalt/ after save       |
| PUCK export              | Manual: download .txt, paste into AI session, verify cold-read |
| API endpoints            | Manual: use browser or curl to hit each endpoint with valid and invalid inputs |
| Visibility filter        | Manual: set flag, filter by Public, confirm <private> rows excluded from export |
| Safe file numbering      | Manual: create two groups in sequence, confirm no number collision |

For V1.1 and beyond: consider Vitest for React component unit tests and
Supertest for backend endpoint integration tests. No coverage target set.

# Future Development (not in V1 scope)

These items are recognized architectural needs. They are not planned for V1.
Document here to prevent re-discovery. Implement when operational need justifies it.

## Encryption layer
PUCKs are portable context artifacts. If exported over untrusted channels or stored
in shared environments, they may contain private or sensitive data. Future need:
public/private key encryption for PUCK export and selective disclosure.
Aligns naturally with the existing Public/Private/Secret visibility model.
Recommended approach when scoped: standard asymmetric encryption (e.g. libsodium)
wrapped as an optional export step in PuckCreator.jsx.

## Server-enforced visibility
Current visibility filtering is UI-enforced only — any client can request any file.
For multi-user or shared deployments, visibility rules must move to the backend:
requests for private/secret files require auth tokens; the backend filters before serving.
Not needed for single-user local deployment.

## Auth layer
No authentication in V1. Single-user, local deployment. If FlowZero is deployed as
a shared or remote service, an auth layer is required before the API contract is
exposed on a network. Suggested: JWT or session tokens per user, scoped to profile group.

## XCML (Extended Context Markup Language)
If CCML Core (7-primitive, hard-capped) is insufficient for a future requirement,
that requirement moves to XCML rather than expanding CCML. XCML is currently an
unspecified reserved namespace. Scope when a concrete requirement exceeds CCML 1.x.

## Predictive Context / Flexicon IDE
The Flexicon functions as a static glossary in V1. Future: when a token is referenced
in a field, the UI surfaces known context from the Gestalt (type, usage, linked profiles,
tags). Makes PUCK creation and narrative generation faster. Requires a lightweight
in-memory index of all token definitions across all loaded profiles.

## Flow Mode (FM)
Named snapshots of active operational context (e.g. "Wednesday mode", "dispatch mode").
Not yet architected. When scoped: a FM is a named set of sticky activations + visibility
presets. Store in FZ_Flow_Config.json or as a separate flow_modes.json.

## Check-In protocol (machine-to-machine)
The current Check-In is human-initiated (paste PUCK into AI session). Future:
a structured protocol for automated PUCK delivery to AI agents or robotic systems.
CCML chunk boundaries (***) already provide the segmentation primitive for this.

### GET /api/list-fz-files
Scans Gestalt/ for files matching the pattern fzXXXX.json (4+ digit number).
Returns the numeric portion of each matching filename as an integer array.
Used by IndexEdit.jsx to determine the next safe profile number.
Degrades gracefully: IndexEdit falls back to index-only count if endpoint is absent.

  Response (200): { "numbers": [0, 1, 2, 3] }
  Error (500):    { "status": "error", "message": "Failed to list files." }

Implementation note: use safeFilePath pattern — only read from GESTALT_DIR, no user-supplied paths.

### POST /api/save-puck ✓ COMPLETE
Writes a compiled PUCK .txt file to public/fz_PUCK_data/.
PuckCreator.jsx POSTs before download; shows ✓/⚠ status message for 12s.
Backup existing file before overwrite (matching save-profile pattern).

  Request body: { "filename": "PUCK_export_fz0001.txt", "content": "<full PUCK text>" }
  filename  required  Bare filename, no path separators, must end in .txt.
  content   required  Full PUCK text content (loader readme + optional flow block + JSON).

  Response (200): { "status": "ok", "message": "PUCK saved to library." }
  Error (400):    { "status": "error", "message": "Missing filename." }
  Error (400):    { "status": "error", "message": "Invalid filename." }
  Error (500):    { "status": "error", "message": "Failed to save PUCK." }

Target dir: path.join(PUBLIC_DIR, 'fz_PUCK_data'). Create dir if absent.

### POST /api/save-puck-index ✓ COMPLETE

  Request body: { "content": "<JSON string>" }

  Response (200): { "status": "ok", "message": "PUCK index saved." }
  Error (400):    { "status": "error", "message": "Missing content." }

# [2026-04-28] MAP — Design Goal Tightened

Munchhausen Architecture Principle design goal (locked):
  "Compress definitions until reconstruction becomes trivial."

This is an optimization target, not just a description. It means:
- If a cold reader (human or AI) must infer anything to reconstruct the system, the artifact is underspecified.
- fz_txt/ is the MAP artifact store. Every non-obvious decision must have a trace here or in config.json.
- Reconstruction requires zero prior context — only assembly.

Apply this criterion when reviewing or updating any fz_txt/ file:
  Q: Can someone with no history rebuild this from only the artifact set?
  If no → compress and specify further.

# [2026-04-28] ServCon — Keep as .txt for Now

ServCon (FZ_01_AI_ServCon.txt) stays as a free-text file intentionally.
Reason: still in learning mode — the right fields, values, and structures are not yet
known. Free text allows discovery without premature schema lock-in.

Future migration trigger: when interaction patterns stabilize and recurring config
combinations emerge (e.g. always using the same mode + rules for coding sessions),
convert to structured JSON config files with selectable values.

Target end state:
  - Named collab profiles as separate JSON files: Playmode.json, Analysis.json, Debug.json, etc.
  - Each file holds a set of ServCon parameters (mode, rules active, signals, escalation level)
  - User picks a profile at session start; system compiles it into the upload-ready .txt format
  - Editor: same Jedit-style panel, no special UI needed

Do not migrate until at least 3-4 stable "modes" have been used repeatedly in practice.
Premature structuring of fluid preference data is a known failure mode.

# [2026-04-28] PUCK Compiler — Future Requirement Note

## Compilation Target Separation

The PUCK compiler currently exports a raw data dump.
Clarification: row-selection IS implemented — the user picks which rows to include.
"Raw data dump" here means the selected rows are output as-is (JSON), not yet shaped
for a specific audience (human vs. AI vs. machine). This is a render target gap,
not a row-selection gap. See FZ_02 Key Concepts for the V1 PUCK definition.

Future requirement: the compiler needs to support named *render targets* — the same
profile data compiled differently depending on intended use.

Examples:
  - target: "ai_prompt"      → renders as behavioral instructions for LLM priming
  - target: "human_brief"    → renders as plain-language summary for a person
  - target: "decision_input" → renders as constraint list for a task (e.g. restaurant selection)

This is a compiler concern, NOT a data model concern.
The underlying profile/token/sticky primitives do not change.
A profile describing AI interaction style and a profile describing a person's dietary
preferences use the same data shape — only the render target differs.

No code change needed now. Revisit when PUCK export is beyond partial status.
